## 一、项目背景与业务目标

### 1.1 背景

当前国内中小学数学作业批改高度依赖教师人工阅卷，以一所 500 人规模的小学为例，数学教师平均每天花费 2-3 小时用于作业批改，且随班级规模增大而线性增长。批改质量也因教师疲劳度、个人风格差异而参差不齐。另一方面，学生在课后做完作业后往往要等到次日才能得到反馈，错误记忆固化的窗口已经错过，形成"做了也白做"的低效循环。

大语言模型（LLM）在自然语言理解方面的突破，使得对结构化文字题目的自动语义评分成为可能；而符号计算引擎（SymPy）对数学表达式的精确计算则弥补了 LLM 在纯数字计算上的幻觉风险。两者结合，可以构建一套"精确 + 智能"的双引擎批改体系，将教师从重复性批改劳动中解放出来，专注于课堂教学设计。

本平台以 **私有化部署** 为核心约束，所有学生数据存储于学校本地，不上传至任何第三方云，满足教育主管部门和家长对数据安全的严格要求。

### 1.2 业务目标

| 编号 | 目标 | 衡量指标 | 目标值 |
|------|------|---------|-------|
| BG-01 | 降低教师批改负担 | 教师日均批改时间 | 2-3 小时 → ≤ 30 分钟 |
| BG-02 | 提升学生反馈及时性 | 提交后获得批改结果耗时 | 次日 → ≤ 3 秒（P95） |
| BG-03 | 保证批改准确率 | MockLLM Harness 准确率 | ≥ 94% |
| BG-04 | 促进学生主动纠错 | Hint 循环后答对率 | ≥ 60%（使用 hint 后） |
| BG-05 | 支持教学决策 | 教师使用班级分析功能频次 | 每周 ≥ 1 次 |

### 1.3 项目范围总览

```
Phase 1（本期）：1-3 年级数学 × 计算题 + 填空题 + 选择题
                私有化单校部署 × 网页端
                AI 批改 + Hint 学习循环 + HITL 人工审核
                Harness CI 质量保障

Phase 2（下学期）：扩展 4-6 年级，支持手写拍照 OCR
                  家长端通知，微信/钉钉推送
                  RAG 题库扩充至万题级别

Phase 3（第二学年）：SaaS 多校版，模型微调，多学科扩展
```

## 四、功能需求

### 4.1 账户与认证

**用户故事**：作为教师，我希望通过账号密码登录系统，以便安全地访问我所在班级的数据。

**验收标准**：
1. 当用户提交正确的用户名和密码时，系统应返回有效的 JWT Token，有效期为 24 小时。
2. 当用户提交错误密码时，系统应返回"用户名或密码错误"提示，且连续失败 5 次后账户锁定 15 分钟。
3. 当 Token 过期时，系统应返回 401 状态码，客户端需重新登录。
4. 系统应支持学生、教师、管理员三种角色，不同角色看到不同的功能入口。
5. 学生账号由教师或管理员创建，不支持学生自主注册。
6. 账户锁定期间，即便输入正确密码也应提示"账户已锁定，请 X 分钟后再试"，不返回"密码正确"信息（防止攻击者枚举）。
7. 登录成功后，系统应在页面顶部显示当前用户名和角色标识，便于共用电脑时确认身份。
8. 用户主动退出登录时，前端应清除本地 Token；后端无需实现 Token 吊销（Phase 1 接受 24h 内旧 Token 可用）。

**边界场景**：
- 用户名长度超过 100 字符 → 返回 422，字段级错误提示
- 密码字段传入 SQL 注入字符串 → 正常返回"密码错误"，无 SQL 异常
- 在账户锁定倒计时结束前 1 秒发起登录 → 仍返回"锁定中"
- 多个浏览器标签页同时用同一账号登录 → 均可正常使用（无 Session 冲突）

### 4.2 作业管理（教师端）

**用户故事**：作为教师，我希望创建一次数学作业并指定题目，以便学生知道要做什么。

**验收标准**：
1. 教师应能创建作业，指定作业标题、所属班级、截止时间和题目列表（从题库选题或手动录入）。
2. 每道题目应包含：题目文本、题目类型（计算题 / 填空题 / 选择题）、参考答案、年级、难度等级（易/中/难）。
3. 系统应支持从题库中按年级、题型、标签筛选选题，也支持教师手动录入新题并同时加入题库。
4. 作业发布后，所属班级的全部学生应能在学生端看到该作业，作业卡片按截止时间升序排列。
5. 教师应能编辑未截止的作业（修改截止时间、增减题目），**已有提交记录的题目不可删除**（系统给出明确提示）。
6. 教师应能查看当前作业的提交进度（已提交人数 / 班级总人数），并可查看哪些学生尚未提交（展示未提交学生名单）。
7. 作业截止时间距现在不足 1 小时时，学生端作业卡片应显示红色"即将截止"标识。
8. 教师应能将一次作业布置给多个班级（一次操作，不需要重复创建）。

**边界场景**：
- 截止时间设置为过去时间 → 返回 422，提示"截止时间必须晚于当前时间"
- 作业题目数为 0 → 不允许发布，提示"至少添加 1 道题目"
- 同一班级同一时间段有 2 个截止时间相同的作业 → 允许（不限制并发作业数）
- 教师删除已被引用的题目 → 提示"该题目已在发布的作业中使用，无法删除"，提供"从作业中移除"入口

### 4.3 答案提交（学生端）

**用户故事**：作为学生，我希望在网页上输入我的答案并提交，以便获得批改结果。

**验收标准**：
1. 学生登录后应看到待完成的作业列表，按截止时间升序排列；已提交的作业单独展示在"已完成"区域。
2. 学生应能对作业中每道题逐题输入文字答案，支持阿拉伯数字、中文数字、分数（如"3/4"或"四分之三"）。
3. 提交前，系统应提示未填写的题目数量，学生可选择"继续提交（空题计错）"或"返回补填"。
4. 点击提交后，系统应在 3 秒内（P95，网络正常）返回批改结果，批改结果以逐题列表形式展示，每题显示：对错标志、AI 反馈文字、鼓励语。
5. 若 AI 批改置信度低于阈值（< 0.85），该题应立即显示"老师正在帮你批改，稍后查看"；前端通过 SSE 订阅审核事件，教师完成审核后自动刷新该题状态。SSE 断线时采用指数退避重连，并提供手动刷新兜底。
6. 截止时间后不允许新提交（返回"作业已截止"提示），但可查看已有批改结果。
7. 同一道题最多允许提交 4 次（hint_level 0-3），第 4 次答错后显示完整解法并锁定提交按钮。
8. 学生可在批改结果页查看每道题的 hint 使用次数和当前状态（待老师审核 / 已批改 / 展示完整解法）。

**边界场景**：
- 学生在截止时间前最后 1 秒点击提交，请求到达服务器时已超时 → 返回"作业已截止"（以服务器时间为准）
- 学生在同一作业重复点击提交按钮（快速连点）→ 只处理第一次请求，后续返回 409 并提示"该作业已提交"
- 学生修改了答案但网络中断，重新连接后页面是否保留已填内容 → 前端应做本地草稿缓存（localStorage），重新加载后恢复
- 答案字段输入超长文本（> 500 字符）→ 前端实时限制输入长度，后端返回 422

### 4.4 AI 批改引擎（核心功能）

**用户故事**：作为学生，我希望提交答案后立即知道对错，并看到适合我年级的中文解释，以便理解自己错在哪里。

**验收标准**：
1. **计算题**：系统应使用 SymPy 符号计算引擎作为精确真值，而非单纯依赖 LLM 判断。SymPy 可以解析如"325+47"、"24×15"、"144÷12"等整数四则运算表达式，精确计算期望答案。
2. **填空题**：系统应对学生答案与参考答案进行规范化后比对，中文数字与阿拉伯数字等价处理（"三百七十二" == "372"），带单位的答案自动去单位后比对（"5元" == "5"）。
3. **选择题**：系统应识别学生填写的选项字母（A/B/C/D，大小写不敏感）与正确选项进行匹配。
4. 当 SymPy 计算结果与 LLM 评估结果一致时，系统批改置信度应 ≥ 0.97（双引擎共识）。
5. 当 SymPy 与 LLM 结论冲突时，系统应以 SymPy 结论为准，置信度降至 0.75，自动进入人工审核队列，入队原因标记为 `sympy_llm_conflict`。
6. LLM 调用失败时，系统应最多重试 3 次，每次重试在 Prompt 中注入上次失败原因；重试耗尽后降级为规则匹配（字符串精确比对），降级批改结果来源标记为 `rule_fallback`，置信度设为 0.80。
7. 批改反馈语言应符合对应年级认知水平：
   - 1-2 年级：极简风格，≤ 2 句，用"你"称呼，词汇小学低年段水平
   - 3-4 年级：先肯定（"你这道题差一点点"），再指导（"在十位进位时…"）
   - 5-6 年级：可使用完整数学术语（"被除数""约分""最小公倍数"等）
8. 系统应对每次批改记录完整的 `agent_trace`（Parser → SymPy → LLM → Router → Classifier → Feedback 各节点的输入输出摘要），教师审核时可展开查看。
9. AI 批改的平均响应时间（从学生提交到返回批改结果）在正常负载（≤ 50 并发）下 P95 ≤ 3 秒。

**边界场景**：
- 学生答案为空字符串 → 直接判错，反馈"你还没有填写答案哦"，不调用 LLM
- 学生答案为纯空格 → 规范化后等同空字符串，同上处理
- 参考答案本身包含中文（如"进位"类填空）→ LLM 专项处理，跳过 SymPy
- SymPy 解析表达式时遇到除以零 → 标记 sympy_success=false，仅用 LLM 判断，置信度按 LLM only 计算
- 题目文本格式为"（  ）+47=372"（括号填空）→ Parser 节点应识别为"求缺失加数"类型
- 学生答案包含 Emoji 或特殊字符 → 规范化过滤后进行比对

### 4.5 Hint 学习循环（Loop 工程）

**用户故事**：作为学生，我希望答错后能得到逐步的提示，而不是直接给出答案，以便我能自己想明白。

**验收标准**：
1. 答错后第一次重提交（hint_level=1），系统应只给方向性提示，**不含答案或数字提示**（如"在进位这里有错，仔细检查"）。
2. 第二次重提交（hint_level=2），系统应指出具体错误在哪一步（如"第一步你算对了，但十位进位时没有加上去"），可给出中间步骤的计算思路，不给出最终答案。
3. 第三次重提交（hint_level=3），系统应给出解题关键提示（如"想一想：个位 7+8=15，要向十位进几？"），此时允许给出关键中间结果，但最终答案仍不展示。
4. 第四次仍答错（hint_level=3 已达上限），系统应展示完整解题步骤（含每步计算结果），并标记该题为该学生的薄弱知识点，提交按钮变为灰色禁用。
5. 某道题连续 2 次达到 hint_level=3 仍答错，系统应向该学生对应班级的教师发送薄弱点预警通知（站内消息）。
6. 答对时，系统应根据是否使用过 hint 给出差异化鼓励：
   - 首次即答对（hint_level=0 → 正确）："一次就答对了，太棒了！"
   - 使用 hint 后答对："经过思考终于答对了，坚持思考很好！"
7. 每次 hint 请求本质上是一次带有更高 `hint_level` 的重新批改请求，系统应生成新的 `grading_result` 记录（`attempt_number` 递增），完整记录每次尝试的输入和批改结果。

**边界场景**：
- 学生直接跳过 hint，在 hint_level=0 答错后立即提交同样错误答案 → hint_level 递增到 1，不循环回 0
- 学生在 hint_level=3 时提交的答案与参考答案等价但写法不同（如"0.5"vs"1/2"）→ 规范化后判正确，不继续 hint
- 两道不同题目的 hint_level 相互独立 → 题 A 的 hint_level 不影响题 B
- 学生退出后重新登录，hint_level 状态是否保留 → 是，从数据库读取当前 attempt_number

### 4.6 人工审核（HITL 队列）

**用户故事**：作为教师，我希望能审核 AI 不确定的批改结果，以便保证最终批改的准确性。

**验收标准**：
1. 系统应将置信度 < 0.85 的批改结果自动推送到对应班级教师的人工审核队列。
2. 教师审核界面应展示（必须展示，缺一不可）：
   - 题目原文
   - 学生原始答案
   - 参考答案
   - AI 批改结论（对/错）及置信度（百分比形式，如"72%"）
   - LLM Chain-of-Thought 推理过程（可折叠展示）
   - SymPy 计算结果（如有）
   - 入队原因（低置信度 / SymPy-LLM 冲突 / 解析失败）
3. 教师应能选择：**通过 AI 结论** / **标记为正确** / **标记为错误**，并可选填覆盖原因；选择"标记为错误"时应同时选择错误类型。
4. 教师覆盖后，学生应在 5 分钟内收到最终批改通知（页面自动刷新对应题目的状态）。
5. 教师覆盖的记录应自动标记为训练样例（`is_training_example=true`），供后续 Prompt 优化使用。
6. 审核队列应支持按以下维度排序：创建时间（最旧优先）、置信度（最低优先）、入队原因（conflict 优先）；默认按创建时间升序。
7. 教师端导航栏应显示待审核数量红点提示，数量 > 99 时显示"99+"。
8. 教师完成所有待审核项后，系统应显示"当前没有待审核的批改结果"空状态页。

**边界场景**：
- 教师在审核过程中网络中断，重新连接后审核结果是否保留 → 前端在提交时做乐观更新，网络恢复后补传，保证不丢数据
- 同一条审核记录被两位教师同时查看（两个标签页）→ 以第一个提交审核结论的教师为准，第二个操作返回"该记录已被审核"
- 审核队列超过 100 条 → 分页展示，每页 20 条，支持翻页和按日期范围筛选
- 教师误操作后能否撤回审核 → Phase 1 不支持撤回，需记录在"reviewer_notes"说明

---

### 4.7 班级与学生分析（教师端）

**用户故事**：作为教师，我希望看到班级整体的错误分布，以便我知道哪些知识点需要在课堂上重点讲解。

**验收标准**：
1. 教师应能查看当次作业的班级统计：平均分（百分制）、各题正确率（条形图）、全班错误类型分布（饼图：计算错误 / 进位错误 / 概念错误 / 审题错误）。
2. 教师应能查看单个学生的批改详情：每道题的对错状态、AI 反馈内容、使用了几次 hint、是否经过人工审核。
3. 系统应展示该学生近 30 天的薄弱知识点列表（按错误频率降序排列，最多显示 5 个）。
4. 当某个知识点班级错误率超过 40% 时，系统应在教师仪表盘的该作业详情页显示"建议课堂重点讲解"黄色提示卡片，点击可查看哪些学生受影响。
5. 数据应支持导出为 Excel 格式：
   - 作业维度：每题的班级正确率、错误类型分布、学生名单
   - 学生维度：某学生所有作业的得分、知识点薄弱程度
6. 教师访问其他班级的学生数据 → 返回 403（服务端校验，不依赖前端控制）。

**边界场景**：
- 作业刚发布，0 人提交时查看统计 → 显示空状态图表，提示"暂无提交数据"
- 班级统计数据在提交后应实时更新（下次刷新时反映最新状态，不做实时推送）
- 某学生近 30 天无错误记录 → 薄弱知识点列表显示"近期表现良好，暂无薄弱知识点"

---

### 4.8 题库管理

**用户故事**：作为教师，我希望维护一个题库，以便每次布置作业时可以复用已有的题目。

**验收标准**：
1. 教师应能新增、编辑、删除题目（删除时检查是否被已发布的作业引用，如有则提示并拒绝删除，提供"软删除从作业中移除"的引导）。
2. 题目应包含以下字段（带 * 为必填）：
   - `*` 题目文本（≤ 500 字符）
   - `*` 题目类型（计算题 / 填空题 / 选择题）
   - `*` 参考答案
   - 解题步骤（可选，供 Feedback 节点参考，JSONB 数组格式）
   - `*` 年级（1-6）
   - `*` 难度（易/中/难）
   - `*` 课程版本（人教版/北师大版）
   - 知识点标签（多选，从预定义标签列表选，或自定义输入）
   - 常见错误答案（可选，附错误类型说明）
3. 系统应支持按年级、题型、知识点标签组合筛选题目，支持按题目文本关键字搜索。
4. 支持批量导入题目（Excel 模板，需提供下载链接），导入时系统校验必填字段，返回逐行的成功/失败报告。
5. 题目导入（或手动创建）后，系统应自动将题目文本向量化（异步任务），存入 Qdrant，供 RAG 相似题检索使用；向量化失败不影响题目的正常使用，仅影响 RAG 推荐质量。
6. 题库应区分"公共题库"（系统内置，所有学校共享）和"校本题库"（本校教师自建），教师搜索时默认展示两者混合结果，可筛选来源。

### 4.9 账户与班级管理（管理员端）

**用户故事**：作为学校管理员，我希望统一管理全校的班级和学生账户，以便学生入学时可以快速开通使用权限。

**验收标准**：
1. 管理员应能创建班级（年级、班级名称、任课教师、学年），一个班级只能对应一名数学教师（Phase 1）。
2. 管理员应能批量导入学生账户（通过 Excel，模板包含必填列：姓名、学号、初始密码、年级、班级名称）；导入后系统返回"成功 N 条 / 跳过 M 条（已存在）/ 失败 K 条（格式错误，附原因）"。
3. 管理员应能将学生分配到对应班级（在导入时指定，或手动调整）；一个学生可属于多个班级（如跨年级选修课）。
4. 管理员应能重置学生或教师的密码（生成临时随机密码并展示，要求用户首次登录修改）。
5. 管理员应能停用/启用学生账户（停用后该学生无法登录，但历史数据保留）。
6. 管理员应能查看全校维度的使用统计（活跃用户数/日、总提交量、AI 批改总数、人工审核率、平均批改准确率）。

**边界场景**：
- 导入 Excel 中的班级名不存在 → 单独报错"班级'X年X班'不存在，请先创建"
- 批量导入 500 条数据 → 应在 30 秒内完成，超过 200 条时使用异步任务，返回 `job_id`
- 管理员误删班级 → 软删除，保留历史作业和批改数据，可在 30 天内从"回收站"恢复

### 4.10 Harness 自动化质量保障（运维端）

**用户故事**：作为系统管理员，我希望每次修改 Prompt 后自动验证批改准确率没有下降，以便安全地发布更新。

**验收标准**：
1. 系统应内置固定的 Phase 1 基线数据集 180 条，覆盖 1-3 年级 × 计算题/填空题/选择题 × 易/中/难三个难度，以及边界用例（中文数字、带单位、等价写法等）；CI 报告必须输出 `total_cases`，少于 180 条直接失败。
2. 当 `prompts/templates/` 或 `agents/nodes/` 目录下任何文件发生变更时，CI 流程应自动触发 Harness 运行（在 PR 合并前执行）。
3. Harness 应支持 MockLLM 模式离线运行（不消耗真实 LLM Token），180 条用例的运行时间 < 60 秒。
4. 当批改准确率 < 94% 时，CI 应标记为失败，阻止本次变更合并，并输出失败用例列表。
5. Harness 运行结果应输出完整报告，包含：
   - 总准确率、误判率（FPR）、漏判率（FNR）
   - 错误分类准确率、置信度校准误差
   - 覆盖矩阵（题型 × 年级 × 难度，每格显示用例数和准确率）
   - 失败用例详情（case_id、预期结论、实际结论、AI 给出的置信度）
6. 管理员应能通过接口手动触发 Harness 运行（支持 MockLLM 和真实 LLM 两种模式），并在 `/api/v1/ops/harness/runs/{run_id}` 查看历史运行记录，保留最近 50 条。

## 五、非功能性需求

### 5.1 性能需求

| 指标 | 要求 | 测量方法 |
|------|------|---------|
| 单次批改响应时间 | P95 ≤ 3 秒（含 LLM 调用，网络正常） | Locust 负载测试，50 并发用户 |
| 批改响应时间（降级模式） | P95 ≤ 0.5 秒（规则匹配降级时） | 单元测试中模拟 |
| 并发批改能力 | 支持 50 名学生同时提交，系统无 5xx 报错 | Locust 50 用户压测 |
| 页面加载时间 | 首屏 ≤ 2 秒（局域网环境，千兆内网） | Chrome DevTools Lighthouse |
| 数据库查询 | 单次查询 ≤ 200ms（P99） | PostgreSQL `pg_stat_statements` 监控 |
| Harness 运行时间 | 180 条用例 MockLLM 模式 ≤ 60 秒 | CI 自动计时 |
| API 非批改接口 | P95 ≤ 500ms（查询类接口） | Locust + 请求日志统计 |

### 5.2 可用性需求

| 指标 | 要求 | 备注 |
|------|------|------|
| 系统可用性 | ≥ 99%（学期内工作日 7:00-22:00，约 1320 小时/学期） | 允许停机 ≤ 13.2 小时/学期 |
| LLM 不可用时 | 自动降级为规则匹配批改，批改功能不中断 | 降级状态下 HITL 队列会增加 |
| 数据库宕机 | 5 分钟内告警，30 分钟内恢复服务 | 通过 Docker healthcheck + cron 监控脚本 |
| 计划内维护 | 仅允许在寒暑假或节假日执行，提前 3 天通知 | 以管理员公告形式通知 |
| 服务重启时间 | 服务重启（含 Prompt 缓存预热）≤ 3 分钟 | Docker compose up 计时 |

### 5.3 安全与合规需求

| 需求 | 说明 | 验证方式 |
|------|------|---------|
| 数据本地化 | 所有学生数据存储在学校本地服务器，不上传至第三方（LLM 调用除外，且调用内容已脱敏） | 网络流量审计 |
| LLM 调用脱敏 | 发送给 LLM 的请求中不包含学生真实姓名、学号等个人标识信息 | 代码审查 + LLM 调用日志核查 |
| PIPL 合规 | 符合《个人信息保护法》，学生数据处理需获得家长授权（学校负责授权收集） | 合规检查清单 |
| 传输加密 | 所有 HTTP 通信使用 HTTPS/TLS 1.2+ | SSL Labs 测试 |
| 密码安全 | 密码使用 bcrypt（cost=12）加密存储，不可逆 | 代码审查 |
| 操作审计 | 教师覆盖批改、管理员账户操作须记录审计日志，保留 1 年，不可删除 | 审计日志查询验证 |
| 角色权限 | 严格 RBAC，学生只能查看自己的数据，教师只能查看本班数据 | 渗透测试：越权访问用例 |
| 输入验证 | 所有用户输入经 Pydantic 校验，防 XSS / SQL 注入 | OWASP ZAP 自动扫描 |

### 5.4 易用性需求

| 需求 | 标准 | 验证方式 |
|------|------|---------|
| 学生端操作步数 | 核心操作（提交答案）≤ 3 步完成（登录 → 选作业 → 输入答案 → 提交） | UAT 观察记录 |
| 教师培训时长 | 无需技术背景，半天培训（4小时）可独立使用全部功能 | UAT 问卷满意度 ≥ 4/5 |
| 移动端适配 | 学生端支持手机浏览器访问（响应式布局，最低兼容 375px 宽度） | Chrome 移动端模拟器测试 |
| 中文界面 | 全部界面使用中文，无英文专业术语暴露给普通用户（技术错误信息中的英文仅出现在 debug 模式） | UI 走查 |
| 无障碍 | 关键按钮和输入框有清晰的文字标签，颜色对比度符合 WCAG AA 标准 | 自动化无障碍扫描工具 |

## 六、约束条件

### 6.1 技术约束

| 约束 | 说明 | 影响 |
|------|------|------|
| **私有化部署** | 不能使用需要互联网持续连接的 SaaS 服务（除 LLM API） | 向量数据库选用 Qdrant（可自部署），不用 Pinecone |
| **Python 生态** | 后端固定使用 Python（LangGraph 生态） | 不考虑 Go/Java 重写 |
| **Docker Compose** | 必须支持 Docker Compose 单命令部署，不引入 K8s | 服务数量和复杂度受限 |
| **OpenAI 兼容 API** | LLM 调用必须兼容 OpenAI Chat Completions 格式 | 可以热切换 DeepSeek/Qianwen/其他兼容模型 |
| **SSE 实时通知（Phase 1）** | 使用 Server-Sent Events，不实现 WebSocket | HITL 和批改状态采用服务端单向推送；断线重连，手动刷新兜底 |

### 6.2 业务约束

| 约束 | 说明 |
|------|------|
| **第一期仅 1-3 年级** | 4-6 年级含分数/百分数，涉及更复杂 SymPy 处理，延后到 Phase 2 |
| **仅数学科目** | 语文、英语等科目的开放性评估超出 Phase 1 范围 |
| **仅数字输入** | 不支持手写作业拍照 OCR（Phase 2 规划） |
| **单校私有化** | Phase 1 不做 SaaS 多租户，架构预留扩展接口 |

### 6.3 外部依赖

| 依赖 | 用途 | 风险 | 缓解措施 |
|------|------|------|---------|
| DeepSeek API | LLM 批改主力 | API 不可用、价格变动 | 备用千问 API；规则降级兜底 |
| Qianwen API | LLM 备用 + Feedback 生成 | 同上 | 切换到 DeepSeek；规则降级 |
| embedding API | 题库向量化（RAG） | 不可用影响 RAG 质量 | 向量化失败静默跳过，不阻断批改 |
| SymPy 库 | 计算题精确验证 | 版本升级 API 变化 | 锁定版本（requirements.txt） |
| Qdrant | 向量检索 | 版本升级 | 锁定版本（docker-compose.yml） |

## 十、验收标准总览

| 模块 | 关键验收标准 | 测试方法 |
|------|------------|---------|
| 账户认证 | JWT 登录正常，角色权限隔离无误，账户锁定机制生效 | API 测试 + 渗透测试 |

## 十一、术语说明

| 术语 | 含义 |
|------|------|
| Hint Level | 学生针对同一道题的第 N 次重提交，级别越高提示越具体（0 = 首次，1-3 = 逐步提示，> 3 = 完整解法已展示） |
| HITL | Human-in-the-Loop，人工介入批改审核闭环，用于低置信度批改结果的人工复核 |
| Harness | 自动化批改准确率测试框架，使用教师标注数据集离线评估 AI 批改质量 |
| SymPy | Python 符号数学库，用于计算题的精确数学真值计算，零幻觉风险 |
| Confidence Score | AI 对本次批改结论的置信度（0.0-1.0），低于 0.85 触发人工审核 |
| agent_trace | 每次批改的完整处理链路记录（各 Agent 节点的输入输出、时间戳、重试次数） |
| Prompt Cache | LLM 服务端对相同前缀 Token 的缓存，降低 Token 消耗约 60% |
| FPR | False Positive Rate，误判率：把正确答案判为错误的比例 |
| FNR | False Negative Rate，漏判率：把错误答案判为正确的比例 |
| RAG | Retrieval-Augmented Generation，检索增强生成，本系统用于从题库检索相似题注入 Prompt |
| tenant_id | 多租户标识，Phase 1 每所学校对应一个 tenant，所有数据按此字段隔离 |
| MockLLM | 用规则/SymPy 替代真实 LLM 的测试模式，零 API 成本，用于 CI 中的 Harness 运行 |
